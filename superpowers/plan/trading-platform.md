# Trading Platform Plan: stock-app → Options/Crypto Trading Bot Platform

## Context

Extend the existing `stock-app` repo (React/Vite frontend on GitHub Pages + FastAPI/`ib_async` backend on a VPS with headless IB Gateway) into a personal trading platform for an options seller. The platform: ingests options strategies from YouTube channels (Theta Profits, Options with Davis) via LLM extraction into a machine-readable strategy database; backtests them (local optopsy engine + an Option Omega manual bridge over its sanctioned CSV channels); runs them as bots on Interactive Brokers (options) and Hyperliquid (crypto perps); tracks performance with roll-chain campaign attribution (Trademetria-style); shows a multi-broker portfolio (IBKR live + Fidelity CSV) with greeks and beta-weighted deltas; provides TradingView-style charts, OptionStrat-style analytics, Option-Alpha-style screeners, and a strategy-aware AI assistant (Claude + Gemini). Deep research (July 2026) identified the open-source components below — **reuse-first, free-first** throughout. This plan is written for incremental implementation by Claude Sonnet, one phase per session or two, in the repo's superpowers plan style (see `superpowers/plan/vix-hedge-screener.md`) with **strict TDD**: every phase writes its tests first (inventory below), and the plan itself is committed to `superpowers/plan/trading-platform.md` as the living source of truth.

## Guiding principles

1. **Reuse first**: extend existing repo modules (listed per section); adopt the vetted OSS below; build only where research confirmed nothing exists (journal roll-chains, Fidelity parser, beta-weighting, spec schema).
2. **Free first**: yfinance/Alpha Vantage/IBKR-owned/DoltHub data by default. Paid options (ThetaData, Tradier, OPRA) only as explicitly-flagged, deferred upgrades. Option Omega is already paid → treat as free.
3. **Provenance everywhere**: every market-data response, extracted rule, and backtest result carries its source (provider + timestamp + delayed/live, or video quote + timestamp, or dataset ref) and the UI displays it.
4. **Safety ladder for real money**: staged orders → paper auto-trade → live warm-up (per-order approval) → live auto-trade within hard risk limits. Safety invariants encoded as tests early so later sessions can't erode them.
5. **User decisions already made** (do not relitigate): extend this repo; SQLite stays (WAL mode); two thin broker adapters, NOT NautilusTrader/LEAN; both backtest engines in one phase; VPS hosting (transcript paste-first); Claude Sonnet + Gemini as the LLM pair; bots end state = auto-trade within limits.

## Architecture

### Database: SQLite stays
Single user, single writer (workers post results over HTTP, never touch the DB file). Enable WAL in `db/session.py` (`PRAGMA journal_mode=WAL`). Volatile shapes go in JSON columns (`spec_json`, bot config, metrics, raw broker payloads) so schema churn lands in Pydantic, not DDL. Additive-only `create_all` migrations as long as possible; add Alembic (`render_as_batch=True`) the first time a column must change. Postgres later = URL swap.

## Strategy spec schema + three compilers

`OptionsStrategySpec` (Pydantic): `meta` (name, channel, video_url) · `universe` (underlyings, sec_type) · `structure: list[LegSpec]` (right, direction, ratio, strike_rule = DeltaTarget|PctOTM|AtmOffset|FixedWidthFromLeg, dte_target/window) · `entry: list[Condition]` · `exit: ExitRules` (profit_target_pct_credit, stop_loss_x_credit, time_exit_dte — each `float | Unspecified`) · `adjustments: list[AdjustmentRule]` (trigger Condition → RollAction: roll_out_same_strike | roll_out_and_away | roll_untested_side | close | convert_to) · `sizing` (bp_pct | fixed_contracts, max_concurrent) · `gates` (VIX regime, earnings avoidance) · `unsupported_conditions: list[str]`.

`Condition.kind` is a closed enum (iv_rank_gte, dte_between, delta_between, vix_below/above, price_above_sma, day_of_week_in, no_earnings_within_days, credit_min_pct_width, funding_rate_gte for crypto, …); every populated field carries `Provenance{quote, timestamp_s}` where LLM-extracted. `Unspecified` is an explicit sentinel — the model must never invent stop losses. Any `Unspecified` in exits or non-empty `unsupported_conditions` ⇒ `needs_review`; approval is always human.

**Interpreter** (`specs/interpreter.py`): `CONDITION_EVALUATORS: dict[kind, async evaluator(params, MarketContext) → CheckResult{passed, observed}]`; `MarketContext` built from dataproviders + analytics (ivrank/ta) — same layers the existing screener uses. `SpecStrategy(Strategy)` wraps approved versions and `build_registry()` appends them, so approved specs appear in the existing `/strategies`, screener, alerts, and order-ticket flow with no new plumbing.

**Compilers** share `CompileResult{config, supported[], unsupported[], warnings}`:
- `compile_backtest` → optopsy legs/filters/PT-SL (conditions the dataset can't express → `unsupported`, shown as "backtest ignores: X" on results) **and** an OO **setup sheet** (rendered list of every OO UI field/value — a run is ~2 minutes of typing) + Custom-Signals CSV (indicator/gate conditions pre-computed locally over daily history — how VIX/MA filters reach OO through its sanctioned upload channel).
- `compile_bot` → strictest: every rule must be in the runtime capability table, every `Unspecified` exit resolved by the reviewer; otherwise bot creation blocked with the exact blocker list. Output `BotConfig` JSON.
- `compile_doc` → markdown doc page: payoff diagram (existing `spread_math` + `PayoffChart`), rule tables with verbatim quote + timestamp link per row, "⚠ not stated in source" for Unspecified, compile-status matrix, claimed-vs-verified performance panel.

## Bot runtime + order safety ladder

**Bots are deterministic rule executors, not LLMs.** No LLM sits anywhere in the trade loop — the LLM only extracts specs (offline, human-approved) and powers the chat assistant (drafts only). A bot is the live execution of the exact same spec a backtest runs: **backtest–live parity is a hard requirement**. If a bot runs Monday–Friday, a backtest of the same spec_version over the same window should reconcile with the bot's realized performance. Mechanisms:
- **One interpreter, two clocks**: `specs/interpreter.py` condition evaluators are pure functions of `(params, MarketContext)`; live ticks build MarketContext from live providers, backtests build it from historical data. No strategy logic exists outside the interpreter — `compile_bot` and `compile_backtest` compile the same rule objects, and any rule one target can't express is a visible `unsupported[]`, never a silent divergence.
- **Parity report** (`backtests/parity.py`, live-enablement phase): for any bot period, auto-run the spec through the local engine over the same dates and produce a trade-by-trade diff (entry/exit dates, strikes, P&L), attributing gaps to fills/slippage/fees vs rule divergence. Rule divergence is a bug; fill divergence is expected and quantified. Surfaced on BotDetail as "live vs backtest".
- Bot decisions are journaled per tick (`bot_run` records verdict + observed condition values), so any trade can be replayed against historical data to verify the rule fired identically.

**Lifecycle**: `draft → running(paper) → live_armed → running(live)`; `paused` resumable; `killed` terminal (new bot required — deliberate friction). Gates: paper requires approved spec + clean `compile_bot`; `arm-live` requires `ALLOW_LIVE_TRADING=true` + live gateway mode + typed bot-name confirmation + ≥1 completed paper round-trip in `bot_run` history.

**Tick loop**: `bot_tick` (RTH, 1 min, reuses `opening_hours.py` extended with `pandas_market_calendars` for holidays/half-days — early closes must shift "manage before close" logic) and `bot_tick_crypto` (24/7). One shared `MarketContext` per underlying per tick (batch quotes; respect IBKR pacing). State machine per bot: FLAT → entry verdict ENTER → **risk gate** → DraftOrder via broker adapter → ladder (below) → ORDER_PENDING → fill event → IN_POSITION → exit/adjustment monitors → close/roll → journal handoff. Bot-originated rolls write `roll_from_structure_id` directly (never rely on the heuristic detector).

**Safety ladder (end state = auto-trade within limits, per user decision):**
1. Manual tickets + assistant drafts: staged `transmit=False` always (existing invariant unchanged).
2. Paper bot: auto-transmits in the paper account when `PAPER_AUTO_TRANSMIT=true` — full autonomy is exactly what paper is for. The adapter asserts paper mode/port before any `transmit=True`.
3. Live bot, warm-up (`approval_policy=warmup_approve`, default for first N trades, N configurable): each order raises `approval_requested` with `expires_at` → ntfy Approve/Reject action buttons → on approve, bot transmits.
4. Live bot, `approval_policy=auto`: bot transmits autonomously **iff** every risk check passes. `transmit=True` exists in exactly one code path (the broker adapter), gated by `(paper mode) OR (live + bot live_armed + ALLOW_LIVE_TRADING + risk gate passed)` — encoded as a unit test in Phase 6 so later sessions can't erode it.

**Risk gate** (`risk.py`, before every order; failure → `risk_blocked` event + alert): whatIf margin vs NetLiq ≤ bot `bp_pct` and global `BOT_MAX_BP_PCT`; per-bot/global max_concurrent; daily realized-loss halt (pauses all bots); per-order max contracts/notional; **portfolio-level caps across ALL bots**: max beta-weighted delta, max net short vega, and a correlated-exposure cap (SPY/QQQ/IWM strangles are one trade in disguise — cap combined BP in underlyings with |pairwise β-correlation| > threshold, using the beta cache); liquidity gate (min OI/volume, max spread width, size vs OI); config sanity bounds (fat-finger guard: sizing outside declared sane ranges requires typed confirmation, and editing a LIVE bot config shows a dry-run diff first). Kill switch (per-bot + global `/bots/kill-all`): cancels working orders, blocks entries, **never auto-flattens** — fires a loud alert with a one-click "draft flatten order". Dead-man: watchdog job alerts on stale heartbeats + optional healthchecks.io ping. Hyperliquid mirrors: testnet↔paper, mainnet↔live; agent wallet is trade-only (cannot withdraw) by construction; single serialized submit queue (nonce safety).

## Backtest robustness: permutation, Monte Carlo, walk-forward

Every backtest with ≥30 trades gets an optional (one-click, on by default for strategy-approval decisions) robustness suite in `backtests/robustness.py` — pure numpy/pandas on the stored trade list / bar data, engine-agnostic (works on optopsy, OO-imported, and manual results alike):

1. **Monte-Carlo permutation test (MCPT)** — "is this edge distinguishable from luck?": build a null distribution by re-running the strategy's entry logic on permuted price data (bar-return shuffling that preserves drift and the distribution but destroys serial structure — Timothy Masters' method; reference implementation `neurotrader888/MonteCarloPermutationTests`, MIT) or, cheaper, by permuting trade entry dates against the same underlying series. Report `p_value = P(permuted metric ≥ actual)` for total return and Sharpe. In-sample p < 0.05 is a gate for promoting a strategy from `backtested` toward paper.
2. **Monte Carlo bootstrap (trade resampling)** — "how bad could the same edge feel?": resample the realized trade P&L sequence with replacement (N=10,000 paths) → distribution of equity curves; report 5/25/50/75/95% terminal-equity bands, max-drawdown distribution (P95 drawdown vs observed), risk-of-ruin at the bot's sizing, and longest-losing-streak distribution. Rendered as a fan chart on the backtest results page; the P95 drawdown feeds the bot's suggested `BOT_DAILY_LOSS_HALT`/BP sizing.
3. **Walk-forward analysis** — "does it survive out-of-sample?": where a spec has tunable parameters (delta target, PT/SL, DTE), rolling anchored windows (default: optimize on 12 months, test on 3, step 3); grid comes from the spec's declared parameter ranges. Report per-window OOS metrics, walk-forward efficiency (OOS/IS return ratio), and the parameter-stability table (does the same neighborhood win each window?). For parameterless specs, degenerate mode = simple OOS holdout splits. Runs in the optopsy worker (it owns the data); OO-imported results get permutation/bootstrap only (entry-date permutation + trade resampling on the imported trade log — full MCPT needs local re-execution).

Where appropriate: permutation + bootstrap on every engine's results; walk-forward only where the local engine can re-execute with varied params. The strategy detail page shows the robustness verdict next to claimed-vs-verified (e.g. "backtest CAGR 18%, MCPT p=0.03, P95 MC drawdown 22%, WF efficiency 0.6").

## oleg_eval — end-to-end parity evaluation harness (development goal loop)

`backend/eval/oleg_eval.py` (CLI: `python -m app.eval.oleg_eval --strategy ma_crossover|zero_dte_ic --report out/`). This is THE acceptance test for the platform's core promise (a bot = a backtest running live) and the goal loop for development: implementation sessions iterate until oleg_eval passes, and it runs in CI from then on. Design:

1. **Fixture strategies** (checked-in spec JSON, exercising both asset classes and the key condition kinds):
   - `ma_crossover`: long SPY (and BTC-PERP variant) when SMA(20) crosses above SMA(50), exit on cross-down. Tests equity/crypto directional legs, indicator conditions.
   - `zero_dte_ic`: sell 0DTE SPX/SPY iron condor with short strikes outside the expected move (`strike_rule: ExpectedMoveOffset`), enter at 11:00 ET (`time_of_day` condition), close at profit target (e.g. 25% of credit) or stop loss (e.g. 2× credit), else expire. Tests multi-leg structure, intraday entry, TP/SL management.
2. **Data compile** (`eval/data.py`): build a deterministic 16-month dataset — real daily/intraday OHLCV from yfinance (cached parquet, pinned date range so the eval is reproducible); for the IC, a **synthetic intraday option chain** generated from the real underlying bars + a Black-Scholes IV model (py_vollib) at 5-minute resolution. Synthetic-but-deterministic is deliberate: free intraday options data doesn't exist, and parity testing needs both engines to consume byte-identical data anyway.
3. **Backtest leg**: run months 1–16 through the local backtest path (compile_backtest → engine) → trade list + equity curve.
4. **Simulated-live leg**: replay months 13–16 through the REAL bot runtime — the actual `bot_tick` state machine, interpreter, risk gate, and a `SimBroker` implementing the `BrokerAdapter` protocol (fills at the same fill model the backtest uses; clock injected/accelerated). No test doubles of strategy logic — only the broker and clock are simulated.
5. **Assertion**: months 13–16 trades from both legs must match trade-for-trade (same entry/exit timestamps at tick resolution, same strikes/sides/quantities) and P&L within a configured fill tolerance (default 0 for the synthetic dataset — identical fill model ⇒ identical numbers). Output: pass/fail + trade diff table + both equity curves. Any mismatch = rule-semantics bug by definition.

Deterministic-clock requirement this imposes (do early, in the paper-bot phase): bot runtime takes an injectable clock/scheduler and consumes MarketContext through the provider interface, so replaying history through the live path is a first-class mode, not a hack. `SimBroker` doubles as the paper-trading fill simulator for Hyperliquid later.

## Journal, portfolio, screeners, assistant specifics

**Roll detection** (`journal/roll_detection.py`): close+open on same underlying within same combo/order ID or same timestamp window ⇒ roll; flat-gap rule (default 3 days, config) breaks chains; auto-detect + **manual merge/split/relink UI** (every override writes a `journal_adjustment` audit row) — the lesson from all commercial tools is that auto-only fails on edge cases. Assignments/exercises/expirations arrive as distinct Flex transaction types and must close campaigns. Full-history first import (FIFO cost basis needs all prior fills).

**Portfolio**: IBKR positions live via `ib_async` (+ account values, margin, model greeks); Fidelity via versioned CSV upload (parser per format notes above; synthetic fixture CSV in tests). Greeks: IBKR model greeks where available, else py_vollib from chain IV. Beta-weighted Δ per tastytrade formula; betas from weekly `beta_refresh` job (1y daily OLS vs SPY, store R², flag low-R² symbols). Grouping: account | underlying | campaign (via journal linkage) | strategy (manual tags for Fidelity). Margin column provider-labeled; Fidelity margin = "unavailable from source". **Risk metrics incl. CVaR** (`portfolio/risk.py`): two complementary measures, both displayed with methodology labels — (a) *realized* VaR/CVaR (95%/99%) of the account equity curve via quantstats (`cvar`/`expected_shortfall`) once journal history exists; (b) *forward-looking* 1-day portfolio CVaR by historical simulation: reprice current positions under the last ~500 trading days' underlying return scenarios (full py_vollib repricing for options — captures gamma, not just delta approximation — with a simple IV shock rule like β·spot-move sticky-IV; linear repricing for stock/perps), take the tail mean of simulated P&L. Non-priceable positions excluded and listed. Per-group CVaR (by underlying/campaign) alongside portfolio total, so concentration is visible.

**Screeners**: generalize `backend/app/screener/` into a registry over the provider layer. Nightly `watchlist_scan` job (chain snapshots rotated through the watchlist within pacing limits) writes `symbol_metrics`; screeners are queries over it: **expensive-options-to-sell** (IV rank, premium yield = credit/BP/day, expected-move vs realized, OI/spread liquidity), high-IVR, 0.16–0.30Δ 21–45 DTE candidates, earnings include/exclude. Rows deep-link to the payoff builder pre-filled and show matching approved strategies. Screeners are assistant tools too.

**GEX module** (`analytics/gex.py` + `dataproviders/cboe_provider.py`): per strike, `netGEX(K) = Σ_exp [Γ_call·OI_call − Γ_put·OI_put] × 100 × S² × 0.01` (dollars per 1% move; standard long-call/short-put dealer convention). Call wall = max positive strike, put wall = max negative; zero-gamma flip = zero-crossing of the signed-GEX profile recomputed over a ±15% spot grid (Perfiliev method); optional BSM vanna/charm per strike in the same loop. Data: CBOE delayed JSON (has gamma/OI/IV per contract; browser User-Agent; cache raw chains to build our own GEX history) with IBKR chain fallback. Refresh 15 min in RTH; walls/flip recomputed after the morning OI refresh. UI: per-strike GEX bar chart centered on spot, flip/wall lines overlaid on the Charts price pane, header tiles (total GEX, distance-to-flip, +γ/−γ regime badge), and displayed caveats (OI is daily → 0DTE positioning invisible; naive dealer assumption ≈ gamma strikes, not MenthorQ's proprietary levels).

**Edge dashboard** (`edge/`, entirely from open data — nothing hand-invented): five signal families per tracked asset (SPX/ES, NQ, VIX, gold, oil, BTC, ETH, watchlist equities where applicable): (1) COT — CFTC Socrata API weekly, net positioning z-scores (1y/3y) per trader class, math ported from kustex/CFTC-COT-Report (MIT); (2) sentiment — CBOE put/call CSVs, CNN F&G, alternative.me crypto F&G, AAII weekly; perps long/short ratio + funding for crypto; (3) macro — fredapi (CPI/PCE, NFP, unemployment, GDP, Fed funds) → regime score; (4) seasonality — in-house monthly mean return + hit rate over 10–15y cached OHLCV; (5) technicals — existing `analytics/ta.py` (price vs 50/200 SMA, RSI, ATR trend). `scoring.py` maps each family to −2…+2 and weight-sums into a per-asset edge score (weights configurable, versioned). Edge page = score table/heatmap with per-component drill-down, every reading provider-labeled. Edge scores also become interpreter condition kinds (`edge_score_gte`, `cot_zscore_gte`) and assistant/screener inputs.

**Assistant** (`assistant/`): Anthropic SDK tool loop (SSE to UI); model routing by role via config: Sonnet for tool-use/analysis, Haiku or Gemini Flash for cheap turns, Gemini for video understanding. Tools: get_positions, get_quotes, get_chain, run_screener, list/get_strategies (specs + completeness + backtests + bot status), run_backtest, get_journal_stats, draft_order (human-confirm only). **Proactive mode**: scheduled `proactive_assistant_scan` compares approved specs' entry conditions against `symbol_metrics`/market state → writes `assistant_suggestion` rows → pushes via existing alert dispatcher (ntfy/email) with deep links. Suggestions are advisory — fully separate from bot execution.

**Frontend**: new pages — Strategies, SpecReview (transcript⇄spec side-by-side, quote highlighting), Extract, Backtests, Bots/BotDetail, Journal/CampaignDetail/JournalStats, Portfolio, Watchlist/Screeners, Charts (lightweight-charts v5: candles + server indicators + IV rank pane + GEX level overlays), Gex (per-strike bars + tiles), Edge (score heatmap + drill-down), OptionChain (source switcher/compare), Assistant (SSE chat + tool transcript + order-confirm modal). ## Risks & mitigations

- **Option Omega coupling**: no automation by design (user decision — brittle + ToS risk). OO participates only through sanctioned CSV channels via the manual bridge; nothing depends on it. optopsy is the programmatic baseline; ThetaData + LEAN/Lumibot is the paid programmatic upgrade if hands-off intraday fidelity is ever wanted.
- **Assignment / early exercise / expiration**: short American options can be assigned any time (ex-dividend dates especially) — positions change without a bot order. Every bot tick starts with **position reconciliation** (broker positions vs expected state); unexplained deltas → bot auto-pauses + alert; Flex assignment/exercise/expiry transaction types close campaigns. Expiring shorts are flattened or rolled per spec before the close — never carried unmanaged into expiration.
- **Bad quotes / order pricing**: limit orders only, priced off mid with a max-slippage band; entries rejected on crossed/no-bid/stale quotes or excessive spread width (a `risk_blocked` event, not a worse fill); bounded repricing loop before giving up.
- **Restart idempotency / duplicate orders**: deterministic client order refs (bot_id + campaign + intent hash); on startup and after any gateway reconnect, `order_record` reconciles against broker open orders BEFORE any tick may act — a restart can never re-send a working order.
- **Gateway downtime blind spots** (IB daily restart, 2FA timeout): no new entries while data/connection is stale; gateway restart scheduled outside RTH; dead-man alert if a bot misses N ticks *while holding positions*. Catastrophe backstop: optional broker-resident protective orders (IBKR GTC brackets, Hyperliquid exchange-side TP/SL) so exits live at the broker even if the VPS dies — mandatory consideration for 24/7 crypto positions held while asleep.
- **VPS exposure + single point of failure**: only 443 published (existing); add API rate limiting and optionally Tailscale/IP-allowlist in front of order-capable endpoints; API token rotation. A documented **manual runbook** (flatten any bot position from TWS mobile / Hyperliquid app when the backend is down) ships with the live-enablement phase.
- **Backups / recovery**: nightly SQLite backup shipped off-VPS (litestream or cron `sqlite3 .backup`) + `.env` escrow; a restore drill is part of the journal phase verification. Parquet caches are re-derivable; the DB is not.
- **Secrets**: all keys stay in `deploy/.env` (never in the public repo or frontend build — existing pattern); the HL agent wallet cannot withdraw by construction; the Flex token is read-only; keep the IBKR API in read-only mode until the live-enablement phase deliberately flips it.
- **Supply chain**: this codebase sits next to money and the ecosystem has had real incidents (pandas-ta repo takeover, malicious LiteLLM releases). Lockfile with hashes (uv/pip-tools), CI installs from lockfile only, renovate/Dependabot with human review, never auto-merge.
- **Prompt injection via ingested text**: transcripts/PDFs are untrusted input. Extraction output is inert, schema-constrained data (closed enums + verbatim-quote guard) reviewed by a human — never executed as instructions; the assistant treats strategy text as data and its only side-effectful tool is human-confirmed `draft_order`.
- **LLM/API cost control**: monthly LLM spend cap (tokens×price summed from `extraction_run`/`chat_message`; batch jobs hard-stop at the cap) alongside the existing `provider_call_log` market-data budget.
- **Margin regime changes**: IBKR raises house margin in vol spikes — entry-time whatIf is not forever. The watchdog recomputes margin cushion on open positions each tick and alerts below a threshold, before IBKR's own liquidation logic can act.
- **ib_async maintenance risk** (community project, ~1 release/yr): version pinned; ibind/Web-API is the documented fallback; all IB imports already funnel through `ibkr/ib_lib.py`, so a swap is contained.
- **Correlation/concentration across bots**: five bots each "within limits" can still be one giant short-vol position. Portfolio-level risk-gate caps (β-weighted delta, net short vega, correlated-BP) are first-class checks, and the Portfolio page's per-underlying CVaR makes concentration visible.
- **Gap risk on short options**: stop losses do not protect through a gap or overnight move — sizing must assume the stop can be jumped. Mitigations: prefer defined-risk structures for early live bots; sizing guidance derived from the MC bootstrap P95 drawdown (not the average); event gates (earnings already in the schema; add FOMC/CPI dates for index strategies from a static schedule).
- **Liquidity exit risk**: a position you can enter but not exit. Entry-side liquidity gate (above) + screener liquidity columns; prefer penny-increment, high-OI chains (SPY/SPX/QQQ tier) for bots.
- **Multiple-testing bias (strategy selection)**: backtest 100 extracted strategies and several look great by luck. Mitigations: MCPT p-value gate; OOS holdout discipline in walk-forward; the strategy DB records how many specs were evaluated (denominator visible next to any winner); mandatory paper period before live regardless of backtest quality.
- **Backtest fill optimism**: mid-price fills (OO and optopsy) overstate live results. Configurable slippage haircut per engine (default: half-spread or a per-contract cent haircut) + the parity/journal loop measures ACTUAL slippage per bot and feeds it back into the assumption; claimed-vs-verified panel shows all three (source claim, backtest, live).
- **Regime dependence of free data**: DoltHub window (2019+) is basically one vol regime plus 2020/2022. Results pages display the tested window prominently; walk-forward windows expose instability; treat pre-2019 claims from videos as unverifiable (labeled).
- **Trading-calendar traps**: holidays, half-days (1pm closes), VIX/SPX settlement quirks — use `pandas_market_calendars` (maintained, Apache-2.0) instead of weekday logic; DTE/expiry math tested against known half-day fixtures (the existing repo already flags AM-settlement issues for VIX).
- **Broker-side independent safety layer**: configure IBKR's own precautionary settings (max order size, max order value, % off-market rejection) in TWS/Gateway config as a second layer the bot CANNOT override, set during the live-enablement phase checklist. The same phase verifies options-trading permission tiers (naked short approval) before a live bot can need them.
- **Notification-channel failure**: a dead ntfy/SMTP = silently blind operator. Critical events (kill, risk_blocked, heartbeat_missed, margin cushion) go dual-channel (push + email); a weekly synthetic "test alert" proves the pipes; healthchecks.io watches the watchdog itself.
- **Provider drift**: yfinance/CNN/CBOE endpoints break without notice. A nightly **canary job** exercises every provider with one known-good request and posts a stale/broken badge + alert on failure — breakage is discovered by the canary, not by a bot mid-trade.
- **Hyperliquid venue risk**: keep only active trading capital on the exchange (bridge the rest out); agent-wallet keys expire — expiry monitored with an alert well before; exchange downtime handled like gateway downtime (no entries, resting exchange-side TP/SL as backstop).
- **Wash sales / taxes**: journal campaigns give the raw data; wash-sale adjustment is explicitly out of scope (IBKR's tax documents remain authoritative) — noted so nobody trusts journal P&L for tax filing.
- **Free options data quality**: parquet-cache once; sanity validators (monotone strikes, put-call parity spot check, missing-day report); dataset_ref + warnings displayed on every result; cross-check a sample month against OO in the backtesting phase.
- **IBKR pacing/subscriptions**: reuse existing chunked chain logic; shared per-tick MarketContext; snapshots not streams for bots; delayed-vs-live flows into provenance so UI can't mislabel; OPRA subscription noted as optional paid upgrade for live options quotes.
- **YouTube blocking from VPS**: paste-first UX is primary by design; structured fallback errors; home CLI; Gemini path is IP-independent (Google fetches the URL).
- **Hyperliquid nonces/rate limits**: one agent wallet, one signer, one serialized submit queue; backoff on 429; persisted fills watermark; testnet mandatory until the Hyperliquid phase verification passes.
- **AGPL (optopsy)**: separate container/pyproject, HTTP/CSV boundary, CI grep forbids `import optopsy` in backend.
- **LLM hallucination**: Unspecified sentinels, verbatim-quote guard (fail → demote to unsupported_conditions), temperature 0, human-only approval, compile_bot re-blocks Unspecified; claims extracted but always labeled as source claims.
- **Safety erosion across many implementation sessions**: invariants as tests from phase 6 (single transmit path, mode gates, lifecycle gates, risk gate) — CI must stay green.
- **SQLite**: WAL; workers write via API only; JSON columns absorb churn; Alembic batch mode is the escape hatch, Postgres the URL-swap last resort.
- **GEX honesty**: naive long-call/short-put dealer assumption (SqueezeMetrics themselves flag its breakdown in stressed markets); OI is daily so 0DTE positioning is invisible; CBOE endpoint may 403 datacenter IPs (browser UA + IBKR fallback + cache). Levels ≈ gamma strikes, not MenthorQ's proprietary models — caveats rendered in the UI, not hidden.
- **Edge data freshness**: COT is weekly (Tue data, Fri release) and CNN/AAII endpoints are unofficial — ingest jobs must fail soft (stale badge on the Edge page, never a crash), consistent with provider-labeling.
- **Corpus copyright**: the McMillan PDF is a copyrighted book sitting in a PUBLIC repo — recommend moving both PDFs out of git history to private storage on the VPS (e.g. a data volume the seed job reads). Extracted rule summaries with short cited quotes are fine for personal use; never expose corpus full text through the public frontend.
